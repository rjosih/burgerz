# Burgerz — Frontend 1.0

Burgerz is Google Reviews for restaurants, with a focus on burger reviews. The UI is mobile-first because most users browse and post reviews from their phones.

This repo contains the frontend only. The backend is out of scope for 1.0 and is replaced by an in-app mock that emulates the eventual API surface. Enjoy hehe

## Mandatory user stories (1.0)

**US-1 — Open a restaurant page**

> As a hungry local, I want to open a restaurant's page and see opening hours and location, so I know I can go there right now.

Acceptance criteria:

- Today's hours highlighted
- "Open now" / "Closed" badge
- Address visible

**US-2 — Submit a review**

> As a burger enthusiast, I want to submit a review with all three scores and a photo, so I can share my opinion.

Acceptance criteria:

- Each of taste / texture / visual scored 1–5 (validated)
- Text up to 500 characters
- Photo optional

## Image delivery

| Concern | Status |
| ------- | ------ |
| Renditions (thumb / medium / large) | Implemented — generated upstream, referenced via `srcset` in [PhotoImage.tsx](src/components/PhotoImage.tsx) |
| CDN delivery | (Future) — `toSource()` currently emits local `/`-prefixed paths only |
| Lazy loading | Implemented — native `loading="lazy"` ([PhotoImage.tsx:34](src/components/PhotoImage.tsx#L34)), eager only when `priority` is set |
| Modern formats | AVIF only (no `<picture>` fallback). Browser support is ~95%+; revisit if the long tail matters |
| Frontend chooses correct rendition | Yes, via `srcset` + `sizes` |

The frontend never processes images. Resize, format conversion, EXIF stripping, and moderation are backend responsibilities (see Scalability and Security below).

## Security

Backend responsibilities (out of scope for the 1.0 frontend, planned):
- Strip EXIF data on upload
- Validate uploads — MIME, magic-byte sniff, dimensions
- Limit file size and type
- Run Rekognition for image and text moderation
- Pre-signed URLs for direct-to-S3 upload

Frontend today:
- Mocked Rekognition flow for visual demo only
- No login, no analytics, no tracking cookies
- Leans on CDN caching 
---

## Scalability

- **Async image processing.** Uploads are stored immediately; heavy work (resize, EXIF strip, moderation) runs as background jobs (Lambda + queue), reporting status back. The API stays responsive, workers scale independently of the API, and failures retry in isolation.
- **No image processing on the frontend.** The frontend handles presentation state; the backend (or its mock) holds data state.
- **Object storage (S3).** Originals and renditions live in S3, served through a CDN at the edge.

## Test strategy

Burgerz follows a risk-based, test-pyramid approach, **integration first**, with unit tests as the wide base and a small Playwright suite at the top for critical journeys.

### Levels

**Unit** — small isolated parts (helpers, formatters, validators, single-component behaviour). Black-box, partition- and rule-driven, sad + happy flows. Authored by the developer who wrote the code. DRY: prefer fewer high-value tests over many similar ones.

**Integration** — the most important level for burgerz. The mandatory user stories depend on multiple parts collaborating (form + state + mock API + UI). Tooling: React Testing Library, Vitest, user event, and the in-app mocks under [src/mocks/](src/mocks/). (Future) MSW could replace the in-app mocks if a network-layer interception story is needed(not installed today).

**System / E2E (Playwright)** — small, focused on highest-risk journeys. Covers page rendering, user flows, form submission, page-level a11y checks, and frontend-to-mock integration in a real browser.

**Acceptance** — does the app meet the user stories? Manual is fine at 1.0 scale; automate with Playwright in CI/CD as the app grows. Decision-table testing is useful here for stakeholder conversations.

### Playwright priorities

| # | E2E test | Reason | Status |
| - | -------- | ------ | ------ |
| 1 | Restaurant page shows opening hours, address, open/closed badge | Mandatory US-1 | Implemented — [e2e/restaurantDetails.spec.ts](e2e/restaurantDetails.spec.ts) |
| 2 | User submits valid review | Mandatory US-2 | (Future) |
| 3 | Invalid review shows validation errors | Protects score/text business rules | (Future) |
| 4 | Failed review submission shows error | Optimistic-UI failure risk | (Future) |
| 5 | Invalid photo upload is rejected | Security / privacy risk | (Future) |
| 6 | Mobile critical flow smoke test | Mobile-first requirement | (Future) |
| 7 | Accessibility smoke test | Frontend quality / WCAG confidence | (Future) |

Deliberately **not** in the Playwright suite (covered by integration / unit instead): opening / closing boundaries, "closed today" branch, full decision-table combinations, full equivalence partitions.

### Test design techniques

- **Equivalence partitioning** — score, text length, file validation
- **Boundary value analysis** — score 1 / 5, text 0 / 500 chars, opening-hour edges
- **Decision tables** — opening-hours and submission rules
- **State transition** — optimistic submission and moderation status

### Coverage

Today, there is no enforced coverage threshold yet. In order to increase quality, we need to set a minimum (and whether high-value features must hit 100%) is a PO/PM + dev/QA decision. Until existing gaps are closed, coverage is **not** part of the pre-push gate, so the gate doesn't block on a debt that pre-existed.

### Other quality work

- (Future) WCAG / Lighthouse, automated and manual occuring 
- (Future) Vulnerability scanning (e.g. Acunetix), static and dynamic
- (Future) OWASP Top 10 alignment, frontend and backend
- (Future) All test levels run on every release across alpha / beta where applicable
- (Future) Quarterly risk analysis + mitigation plan; periodic penetration testing (the app is too small to pen-test every release)
